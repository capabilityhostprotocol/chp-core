# 0031: Mutual TLS for the HTTP Transport

- **Status:** shipped (spec v0.8.1, chp-core 0.39.0, npm alpha.30)
- **Issue:** rad:a5db8147
- **Affects:** chp-v0.2.md §5 (a third caller-auth credential) + chp-http-binding.md §2
  (the mTLS rule). **No new object, no new reserved code** — mTLS is a transport-layer
  option; a failed handshake is a connection-level refusal. Additive: a host without
  TLS configured is byte-identical. Spec **v0.8.0 → v0.8.1** (a patch).

## Problem

0027 made TLS a **MUST** (§5) and added ed25519-signed bearer tokens, but left
**mutual TLS unspecified** — a caller could not authenticate with a client certificate,
and a host had no defined way to bind a verified client-cert identity to evidence. In
a private mesh with an existing PKI (or a service mesh issuing workload certs), mTLS
is the natural caller credential: authentication happens at the transport handshake,
before any application byte, and the same certificate that encrypts the channel
identifies the caller.

## Design

**Server (stdlib `ssl`, no new dependency).** `create_http_server` /`serve_http` gain
`certfile` / `keyfile` / `cafile`. When `certfile` is set the server wraps its
listening socket in an `ssl.SSLContext`; adding `cafile` sets `verify_mode =
CERT_REQUIRED` + `load_verify_locations(cafile)`, so every accepted connection must
present a client certificate the CA signed — an unknown-CA or absent cert is refused
**at the handshake** (no bytes reach a handler, no HTTP response, no reserved code).

**Identity binding.** On a verified connection the handler reads the peer certificate
(`getpeercert()`) and derives the caller identity — subject **commonName**, else the
first **DNS subjectAltName**. This becomes the verified evidence `subject`
(`{id, type: "mtls", verified: true}`), overriding any client-asserted subject exactly
as a key or token does — a **third** caller-auth credential beside `X-CHP-Key` and
`X-CHP-Token`, and, because the TLS layer already authenticated it, checked **first**.

**Client.** `RemoteCapabilityHost` gains `client_cert` / `client_key` / `cafile`; over
`https` it builds a client-cert `SSLContext` and presents it on every connection.

**Honest boundary.** CHP specifies *how a verified client cert binds to evidence*, not
a PKI. Certificate issuance, rotation, revocation, and CA operation are deployment
concerns, deliberately out of scope — mTLS satisfies §5's TLS-MUST and caller-auth
MUST simultaneously; it does not define how the certs come to exist.

## Compatibility

Additive. A host started without `certfile` serves plain HTTP exactly as before; a
client without `client_cert` connects exactly as before. No wire object changes, no
new reserved code (a handshake failure is connection-level, not a `DenialReason`). The
only evidence change is a new `subject.type` value `"mtls"` for mTLS-authenticated
callers (the existing `"api_key"` is unchanged). A **patch** bump (v0.8.1).

## Deferred by design

Certificate **issuance / rotation / revocation** and CA operation (a PKI — out of
scope); **SPIFFE/SVID** identity mapping; cert-bound **capability scope** (an mTLS
caller currently maps to a subject, not a scoped key — a scope binding could layer on
later); OCSP/CRL freshness; combining an mTLS identity *and* a bearer token on one
request (mTLS wins when present).
